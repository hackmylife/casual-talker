## 1. Go移行のビジネスインパクト

**判定: 許容する。**

+7日（11日→18日）の増加は以下の理由で妥当:
- Go側でSSEストリーミング、JWT検証、OpenAI SDK統合をゼロから構築

- フロント/バック分離によるCORS、デプロイパイプライン、環境変数管理の二重化

**18日は上限。これ以上の延伸は認めない。**

---

## 2. フロント/バック分離に伴う複雑化

**判定: 許容範囲。**

増加する複雑性:
- デプロイ先が2箇所（Vercel + Cloud Run）
- CORS設定、認証トークンの受け渡し
- ローカル開発環境でフロントとバックを同時起動

**条件: Phase 0でdocker-compose.ymlによるローカル一発起動を整備すること。**

---

## 3. Cloud Runの選定

**判定: 適切。**

- Goシングルバイナリとの相性が良い
- 従量課金でMVPコスト最小化
- 将来のスケールアウトも容易

**条件: min-instances=1を必須。レイテンシ目標3秒はウォームインスタンス前提。**

---

## 4. sqlc + gooseの選定

**判定: 適切。MVPに最適。**

- sqlcはSQL直接記述 + 型安全Goコード生成。セキュリティ監査が容易
- RLSバイパスしてGo側でWHERE user_id制御は合理的
